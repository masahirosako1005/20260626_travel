# 観光再訪アプリ「旅のつづき」MVP仕様書

- 作成日: 2026-06-26
- 版: 1.0
- 対象: ローカルPC上で動作するWebモック版
- 実行形態: 1台のPCでNode.js Webサーバーを起動し、同一PCのブラウザからアクセスする
- 想定技術: Node.js / Express / SQLite / HTML / CSS / JavaScript

---

## 1. 目的

本アプリは、観光地への来訪後に蓄積される行動データを起点として、ユーザーごとに「再訪する理由」を提示する観光アプリである。

既存のスポット巡り、スタンプ獲得、クーポン、SNS投稿機能を拡張し、観光アプリを「一度きりのイベント」から「継続接点」へ発展させる。

### 1.1 達成したいこと

- 観光客全般に対して、まち歩き・スポット巡りの体験価値を提供する
- 一度来訪した観光客に対して、未完了・思い出・嗜好に基づく再訪理由を提示する
- 地元住民・近隣住民に対して、地域再発見・短時間回遊・季節イベント参加のきっかけを作る
- 地域・運営者に対して、人流、滞在、スタンプ取得、特典利用、推薦ルート利用状況を可視化する
- モック版では、1台のPC上で開発・デモ・検証できる軽量構成とする

---

## 2. 背景・設計方針

デジタルスタンプラリーは、QRコードによるスポットチェックイン、特典交換、抽選、イベント回遊促進などで広く使われている。一方で、一般的なスタンプラリーはイベント期間中の単発利用で終わりやすい。

本アプリでは、来訪後の行動データを蓄積し、以下の観点で再訪施策へ変換する。

1. 思い出し: 前回の訪問スポット、取得スタンプ、利用特典、滞在体験を振り返る
2. 未完了感: 未取得スタンプ、未訪問スポット、未完了ラリーを提示する
3. 個別動機: 履歴、嗜好、関与度に応じて、次に訪れる理由を提示する
4. 地域愛着: 地域との関係性をスコア、称号、レベル、バッジで表現する
5. 管理改善: 管理者が人気スポット、離脱スポット、回遊順、推薦利用率を確認し、施策を改善する

## 7. 機能要件

## 7.1 JSONインポート機能

### 概要

管理画面からJSONファイルをアップロードし、ユーザー、スポット、ラリー、特典、訪問履歴、スタンプ取得、特典利用、推薦ルート利用履歴をSQLiteへ登録する。

### 入力

- JSONファイル
- 文字コード: UTF-8
- 形式: 本仕様書の「14. サンプルJSON仕様」に準拠

### 処理

1. JSON構文チェック
2. 必須キー確認
3. ID重複確認
4. SQLiteへトランザクション登録
5. 取り込み結果を表示

### 出力

- 登録件数
- エラー件数
- エラー詳細

---

## 7.6 未完了一覧機能

### 概要

未取得スタンプ、未訪問スポット、未完了ラリーを表示する。

### 判定条件

- ラリー対象スポットのうち、スタンプ未取得のスポットがある
- 登録スポットのうち、訪問履歴がないスポットがある
- ラリー完了率が100%未満

### 表示内容

| 表示項目 | 説明 |
|---|---|
| 未完了ラリー名 | 完了していないラリー |
| 残りスタンプ数 | 完了までの必要数 |
| 未取得スポット | 取得対象スポット |
| 推定所要時間 | 標準滞在時間から算出 |
| おすすめ理由 | 再訪を促す説明文 |

## 7.7 個別おすすめルート機能

### 概要

ユーザーの訪問履歴、スタンプ取得、滞在時間、通過時間、特典利用履歴をもとに、次回訪問向けのルートを提示する。

### 対象ルート形式

- 60分コース
- 90分コース
- 半日コース
- 1日コース
- 季節イベントコース
- 地元再発見コース

### 推薦スコア

スポットごとに以下のスコアを算出する。

```text
recommend_score =
  unvisited_score
+ unacquired_stamp_score
+ preference_category_score
+ long_stay_category_score
+ coupon_category_score
+ rally_completion_score
+ season_event_score
- visited_penalty
- pass_only_penalty
```

### スコア例

| 条件 | 加点/減点 |
|---|---:|
| 未訪問スポット | +20 |
| 未取得スタンプあり | +30 |
| 長時間滞在カテゴリと一致 | +15 |
| 特典利用カテゴリと一致 | +10 |
| あと1〜2個でラリー完了 | +25 |
| 季節イベント対象 | +10 |
| 既訪問済み | -10 |
| 前回通過のみ | -5 |

### ルート生成ルール

1. ユーザーごとに各スポットの推薦スコアを計算する
2. 未取得スタンプを優先する
3. ユーザー嗜好カテゴリと一致するスポットを優先する
4. 各コースの時間枠に収まるようにスポットを選ぶ
5. 同一エリア内または隣接エリアを優先する
6. 特典が利用可能なスポットを少なくとも1件含めることを優先する

## 7.8 ユーザータイプ分類

### 概要

スコアリングと簡易クラスタリングにより、ユーザーを分類する。

MVPでは、実装容易性を優先し、まずはルールベース分類を採用する。将来、K-means等のクラスタリングへ置き換え可能にする。

### タイプ

| タイプ | 判定例 | 提案内容 |
|---|---|---|
| コンプリート型 | スタンプ取得率が高い | 未取得スタンプ、ラリー完了、称号 |
| じっくり滞在型 | 平均滞在時間が長い | 文化、体験、温泉、展示 |
| 回遊型 | 訪問スポット数が多い | 複数スポットを巡る効率ルート |
| グルメ・特典型 | 特典利用が多い | グルメ、買い物、クーポン付きルート |
| 再発見型 | 地元/近隣住民、短時間利用 | 季節イベント、穴場、地元おすすめ |

## 7.9 地域愛着スコア機能

### 概要

ユーザーと地域との関係性をスコア、レベル、称号、バッジ、コメント、進捗バーで表現する。

### 算出式

```text
attachment_score =
  visit_spot_count * 8
+ total_stay_minutes * 0.2
+ acquired_stamp_count * 10
+ benefit_use_count * 8
+ completed_rally_count * 20
+ revisit_count * 25
```

100点を上限とせず、継続的に伸びるスコアとして扱う。

### レベル

| レベル | スコア範囲 | 称号 |
|---|---:|---|
| Lv.1 | 0〜49 | はじめての旅人 |
| Lv.2 | 50〜99 | まち歩き見習い |
| Lv.3 | 100〜179 | 地域探索者 |
| Lv.4 | 180〜299 | まちの常連 |
| Lv.5 | 300以上 | 地域アンバサダー |

### バッジ例

- 歴史スポット好き
- 自然満喫派
- グルメ開拓者
- 特典活用上手
- ラリーコンプリート
- 地元再発見マスター

## 9.2 DDL案

```sql
CREATE TABLE users (
  user_id TEXT PRIMARY KEY,
  nickname TEXT,
  user_segment TEXT,
  created_at TEXT,
  analytics_consent INTEGER DEFAULT 1,
  nickname_display_consent INTEGER DEFAULT 1
);

CREATE TABLE spots (
  spot_id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  category TEXT NOT NULL,
  area TEXT NOT NULL,
  avg_minutes INTEGER DEFAULT 30,
  description TEXT,
  image_path TEXT
);

CREATE TABLE rallies (
  rally_id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  season_tag TEXT
);

CREATE TABLE rally_spots (
  rally_id TEXT NOT NULL,
  spot_id TEXT NOT NULL,
  PRIMARY KEY (rally_id, spot_id),
  FOREIGN KEY (rally_id) REFERENCES rallies(rally_id),
  FOREIGN KEY (spot_id) REFERENCES spots(spot_id)
);

CREATE TABLE benefits (
  benefit_id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  category TEXT NOT NULL,
  spot_id TEXT,
  estimated_value_yen INTEGER DEFAULT 0,
  valid_from TEXT,
  valid_to TEXT,
  FOREIGN KEY (spot_id) REFERENCES spots(spot_id)
);

CREATE TABLE visits (
  visit_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  session_id TEXT NOT NULL,
  spot_id TEXT NOT NULL,
  visited_at TEXT NOT NULL,
  route_order INTEGER,
  stay_minutes INTEGER DEFAULT 0,
  pass_minutes INTEGER DEFAULT 0,
  event_type TEXT CHECK(event_type IN ('stay','pass')),
  FOREIGN KEY (user_id) REFERENCES users(user_id),
  FOREIGN KEY (spot_id) REFERENCES spots(spot_id)
);

CREATE TABLE stamps (
  stamp_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  session_id TEXT NOT NULL,
  spot_id TEXT NOT NULL,
  rally_id TEXT,
  acquired_at TEXT NOT NULL,
  method TEXT DEFAULT 'mock_button',
  FOREIGN KEY (user_id) REFERENCES users(user_id),
  FOREIGN KEY (spot_id) REFERENCES spots(spot_id),
  FOREIGN KEY (rally_id) REFERENCES rallies(rally_id)
);

CREATE TABLE benefit_uses (
  benefit_use_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  session_id TEXT NOT NULL,
  benefit_id TEXT NOT NULL,
  spot_id TEXT NOT NULL,
  used_at TEXT NOT NULL,
  estimated_value_yen INTEGER DEFAULT 0,
  FOREIGN KEY (user_id) REFERENCES users(user_id),
  FOREIGN KEY (benefit_id) REFERENCES benefits(benefit_id),
  FOREIGN KEY (spot_id) REFERENCES spots(spot_id)
);

CREATE TABLE route_usages (
  route_usage_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  session_id TEXT NOT NULL,
  route_id TEXT NOT NULL,
  viewed_at TEXT NOT NULL,
  used INTEGER DEFAULT 0,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
);

CREATE TABLE recommendations (
  recommendation_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  route_type TEXT NOT NULL,
  score REAL NOT NULL,
  reason TEXT,
  generated_at TEXT NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

| GET | `/api/admin/revisit-candidates` | 再訪候補一覧 |
| GET | `/api/admin/analytics/routes` | 回遊順分析 |

## 11. 推薦ロジック詳細

## 11.1 嗜好カテゴリ推定

ユーザーごとにカテゴリスコアを算出する。

```text
category_score =
  visit_count * 5
+ total_stay_minutes * 0.2
+ stamp_count * 4
+ benefit_use_count * 8
- pass_count * 2
```

最大スコアのカテゴリを主嗜好カテゴリとする。

## 11.2 未完了ラリー判定

```text
completion_rate = acquired_stamp_count_for_rally / target_spot_count_for_rally
```

- 100%: 完了
- 67%以上100%未満: コンプリート目前
- 34%以上67%未満: 進行中
- 34%未満: 未着手に近い

## 11.3 ルート時間制約

| ルート | 目安時間 | スポット数 |
|---|---:|---:|
| 60分コース | 60分 | 1〜2 |
| 90分コース | 90分 | 2〜3 |
| 半日コース | 180〜240分 | 3〜5 |
| 1日コース | 360〜480分 | 5〜8 |
| 季節イベントコース | イベント設定に依存 | 2〜5 |
| 地元再発見コース | 60〜120分 | 2〜4 |

## 12. KPI定義

## 12.1 再訪率

```text
再訪率 = 2回以上セッションがあるユーザー数 / 全ユーザー数
```

## 12.2 スタンプ完了率

```text
スタンプ完了率 = 完了ラリー数 / ユーザー参加ラリー数
```

## 12.3 平均訪問スポット数

```text
平均訪問スポット数 = stayイベント数 / セッション数
```

## 12.4 推薦ルート利用率

```text
推薦ルート利用率 = used=trueの推薦ルート数 / 推薦ルート閲覧数
```

## 12.5 地域内消費額推定

実決済データは扱わないため、MVPでは以下で推定する。

```text
地域内消費額推定 = 特典利用回数 × 特典ごとの想定価値
```

この値は実売上ではなく、消費関与の代替指標として扱う。

## 13. 権利・著作権・プライバシー方針

## 13.1 個人情報

- ユーザー識別は匿名IDを基本とする
- 利用者画面ではニックネームを表示する
- 管理画面では匿名IDとニックネームを表示するが、メールアドレスや電話番号は扱わない
- モックデータはすべて架空データとする

## 13.2 SNS

- MVPではSNS投稿本文、画像、アイコンを保存しない
- 投稿有無、投稿数などのメタ情報のみ将来拡張対象とする
- 外部SNSの本文・画像を無断保存しない

## 13.3 画像

- モックでは自作素材、権利許諾済み素材、またはプレースホルダー画像を使用する
- 観光協会、自治体、店舗素材を利用する場合は、使用許諾と表示条件を確認する

## 13.4 地図

- MVPでは地図機能を必須としない
- 将来OpenStreetMap系を利用する場合は、利用規約、タイル利用条件、著作権表示を確認する
